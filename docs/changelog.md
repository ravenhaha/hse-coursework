# Changelog — История изменений модели данных

Здесь фиксируются все изменения схемы БД и domain-модели,
чтобы было видно, что и почему менялось.

## [v0.5] — Теги: полноценная реализация

**Дата:** _подставь сегодняшнюю_

### Реализовано (без изменений схемы — структура Tag была заложена в v0.1)

- **CRUD тегов**: создание / переименование / удаление, всё в рамках юзера.
- **Уникальность имени тега в рамках юзера** — `UniqueConstraint(user_id, tag_name)`.
- **Привязка тегов к материалам** через M2M-таблицу `material_tags`
  с каскадным удалением.
- **Три варианта работы со связями:**
    - `POST /tags/materials/{id}` — добавить один тег (one-shot);
    - `DELETE /tags/materials/{id}/{tag_id}` — убрать один тег;
    - `PUT /tags/materials/{id}` — bulk-замена всего набора.
- **Дефолтные теги** при регистрации (`лекция`, `семинар`, `домашка`,
  `экзамен`, `конспект`, `формула`, `важно`, `повторить`). Реализованы
  как обычные теги юзера с `user_id` — он их может удалять и
  переименовывать как угодно.

### Защита от race condition

Все операции, нарушающие `UniqueConstraint`, защищены через
`try/except IntegrityError` поверх предварительной SELECT-проверки.
Это страхует от двойного клика / параллельных табов.

### Решено НЕ добавлять (пока)

- **Цвет тега** — добавим, когда фронт скажет, что нужно. Простая
  миграция на одно поле `color: String(7)`.
- **Глобальные shared-теги** — отказались в пользу seed'а персональных
  при регистрации. Юзер должен иметь полный контроль над своими тегами.
- **Иерархия тегов / категории тегов** — overkill для MVP.

## [v0.4] — Иконки коллекций и реализация Auth-стека

**Дата:** _подставь сегодняшнюю_

### Добавлено

- **`Collection.icon`** — emoji-иконка коллекции (`String(10)`, nullable).
  Хранится как строка с эмодзи (например, `📚`, `🧪`). Длина 10 символов —
  с запасом на составные эмодзи (zwj-последовательности, модификаторы кожи).
  Альтернатива (загружаемая картинка) **отклонена**:
    - 0 байт на диске vs сотни KB на каждую коллекцию
    - мгновенный рендер без HTTP-запросов
    - стандарт UX в подобных продуктах (Notion, Linear, ClickUp)

### Реализовано (без изменений схемы)

- **Слой аутентификации полностью готов**:
  локальная регистрация (email + password с автологином),
  локальный логин, refresh-цикл, OAuth (Yandex реализован,
  VK — заготовка под токены).
- **HttpOnly-cookie стратегия**: access + refresh + csrf в трёх отдельных
  cookie. Фронт не видит токены через JS, защита от XSS.
- **CSRF double-submit pattern**: `csrf_token` доступен для чтения JS,
  фронт обязан слать его в заголовке `X-CSRF-Token` на mutating-запросы.
- **Argon2** для паролей с авто-rehash при ужесточении параметров.
- **Дефолтные теги** создаются при первой регистрации (любым способом).

### Решено НЕ добавлять

- **Загружаемая обложка коллекции** — см. выше, emoji эффективнее.
- **Поле `notes`/`description` у материала** — для комментариев к файлу.
  Юзер может создать рядом текстовый материал в той же коллекции,
  если нужно прокомментировать. Добавим, если фича понадобится по фидбеку.
- **Batch-upload эндпоинт** — фронт делает N запросов к одиночному
  upload-эндпоинту. Это даёт независимые транзакции, частичный успех
  и прогресс по каждому файлу.

## [v0.3] — Материалы: метаданные файлов и поиск по содержимому

**Дата:** _подставь сегодняшнюю_

### Добавлено

- **`Material.file_size`** — размер файла в байтах (`BigInteger`, `NULL` для текстов).
  Нужно для отображения в UI («PDF, 2.4 MB») и для будущих лимитов хранилища.
- **`Material.extracted_text`** — текст, извлечённый парсером из PDF/DOCX/etc.
  Хранится отдельно от `text_content` (который — содержимое текстовых заметок).
  Используется для **полнотекстового поиска по содержимому файлов**.

### Изменено

- **CHECK constraint `source_content_match`** — расширен под новые поля:
  - Для `source_type='text'`: `text_content` обязателен, все file-поля = `NULL`.
  - Для `source_type='file'`: `file_path` + `file_size` обязательны,
    `extracted_text` опционален (может быть `NULL`, если парсер не сработал),
    `text_content` = `NULL`.
- **Поиск материалов** (`search_materials`) теперь ищет подстроку в:
  `material_name` OR `text_content` OR `extracted_text`.

### Решено НЕ добавлять

- **`Material.original_name`** — оригинальное имя загруженного файла.
  Изначально планировалось, но решили обойтись одним `material_name`:
  - Юзер всё равно переименовывает материал под себя.
  - При скачивании используем `material_name + расширение` — это даёт
    юзеру **то имя файла, которое он хочет**, а не загруженное случайно.
  - Меньше полей — проще модель.
- **`Material.mime_type`** — для MVP не нужен. Расширение в `file_path` —
  достаточный источник истины для текущей логики.

### Отложено в roadmap

- `User.plan` — тариф (free / pro)
- `User.storage_used` — занятое место в байтах (агрегируется по `file_size`)
- Полнотекстовый индекс PostgreSQL (`tsvector`) — пока хватает `ILIKE`

## [v0.2] — ER-диаграмма (логическая модель)

**Дата:** _подставь сегодняшнюю_

### Добавлено

- **Таблица `Auth_account`** — для поддержки OAuth (VK, Yandex).
  Один юзер может иметь несколько способов входа.
  Структура: `user_id`, `provider` (email / vk / yandex), `provider_user_id`
  (для email = сам email; для OAuth = id у провайдера), `password_hash`
  (только для provider=email; для OAuth = NULL).
  `UNIQUE (provider, provider_user_id)` гарантирует, что один VK-аккаунт
  не может быть привязан к двум юзерам.
- **`User.is_active`** — флаг активности (для soft-блокировки аккаунта).
- **`User.updated_at`** — дата последнего изменения профиля.
- **`User.avatar_url`** — путь к загруженному аватару (опционально).

### Изменено

- **`User.hashed_password` УБРАН из модели User**.
  Пароль теперь живёт в `Auth_account.password_hash` только для
  записи с `provider='email'`. Это позволяет юзеру:
    - иметь только OAuth-вход без пароля
    - иметь email+пароль + дополнительно несколько OAuth
  Без дублирования и без NULL-костылей в `User`.

### Исправлено

- Опечатка в `Auth_account`: колонка называется `provider`, а не `provider_auth`.
- `UNIQUE (provider, provider_user_id)` — теперь корректно.

### Отложено в roadmap (не входит в MVP)

Решено пока **не добавлять** эти поля, чтобы не усложнять MVP.
Добавим, когда дойдём до фичей лимитов и тарифов:

- ~~`Material.file_size`~~ — **добавлено в v0.3**
- `Material.mime_type` — MIME-тип файла
- `Material.is_important` — флаг «Важное»
- `User.plan` — тариф (free / pro)
- `User.storage_used` — занятое место в байтах
