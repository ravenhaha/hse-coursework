> Документ описывает суть приложения, его пользователей, основные сценарии

## Блок 1. Суть приложения

### 1.1. Что это за приложение?

«Омут памяти» — веб-приложение для учащихся, которое помогает хранить,
структурировать и визуализировать учебные материалы (заметки, документы,
изображения) и быстро возвращаться к ним при подготовке к экзаменам.

### 1.2. Какую боль решает?

**Боль:** перед экзаменом или при возврате к теме студент тратит часы,
чтобы собрать материалы, разбросанные по закладкам браузера, скачанным
файлам, конспектам в тетради и заметкам в Telegram.

**Решение:** «Омут памяти» собирает всё в одном месте, даёт удобную
иерархию коллекций и граф связей между материалами — чтобы восстанавливать
контекст за минуты, а не за часы.

### 1.3. Чем отличается от аналогов?

Сравнение с Notion, Obsidian, Pocket, Evernote:

- **Заточен под учёбу**, а не под всё подряд (как Notion). Интерфейс
  и сценарии оптимизированы под повторение материала.
- **Граф связей "из коробки"** — не нужно настраивать как в Obsidian,
  всё работает сразу.
- **(в будущем) AI-ассистент** — автоматически генерирует теги, краткие
  пересказы, тестовые вопросы по материалу для самопроверки, делит текст
  на чанки и строит граф по смысловым связям, а не по файловой структуре.
- **Размещён в РФ** → работает без VPN, оплата российскими картами.

## Блок 2. Что хранит

## 2.1. Сущности приложения

На MVP — **5 основных сущностей** + 1 связующая таблица:

1. **User** — пользователь системы
2. **Collection** — коллекция (папка) с материалами (плоская, без вложенности
   на MVP). Имеет **emoji-иконку** (`icon`) для визуальной идентификации в UI —
   по аналогии с Notion / Linear. Хранится как строка-эмодзи, не как файл.
3. **Material** — единица контента (текст или файл)
4. **Tag** — пользовательский тег для материалов
5. **Auth_account** — привязка способов входа (email/пароль, VK, Yandex)
   к юзеру. Один юзер может иметь несколько способов входа.

**Связующая таблица:**
- **Material_Tag** — M:N между Material и Tag (без дополнительных полей).

### 2.2. Что отложено в roadmap

На MVP больше ничего не храним. В roadmap могут появиться:

- **UserSettings** — индивидуальные настройки (тема, язык интерфейса)
- **ActivityLog** — история действий для аналитики и восстановления
- **MaterialLink** — ручные связи между материалами для графа
- **RefreshToken** — если потребуется отзыв токенов

## Блок 3. Кто пользуется

### 3.1. Типы пользователей

На MVP — **один тип**: зарегистрированный пользователь.

В roadmap:
- **Гость** — read-only по публичным коллекциям
- **Администратор** — модерация, аналитика

### 3.2. Личное или совместное?

**MVP — строго личное хранилище**: каждый видит только свои данные.

В roadmap — шеринг коллекций по ссылке (read-only / collaborative).

### 3.3. Публичный режим?

Публичного режима **нет**. Любая страница, кроме `/login` и `/register`,
требует авторизации.

## Блок 4. Главные действия (use cases)

### 4.1. Сценарий нового пользователя

1. Регистрация → вход
2. Welcome-экран: краткая инструкция + CTA «Создать первую коллекцию»
3. Создание коллекции (например, «📚 Биология»):
   - юзер вводит название
   - опционально выбирает эмодзи-иконку из пикера (по умолчанию — без иконки)
4. Добавление материала в коллекцию:
   - **Вариант А:** загрузка файла с компьютера (PDF / картинка / txt)
   - **Вариант B:** создание заметки прямо в приложении
   - **Bulk:** юзер может в одной модалке выбрать N файлов — фронт
     отправляет N отдельных запросов, показывая прогресс по каждому
5. Опционально: проставить теги, отметить как «Важное»
6. Сохранение → материал появляется в дереве сайдбара
7. Поиск:
   - по всем материалам — подстрока в названии / тексте заметки /
     извлечённом тексте файла; фильтры: коллекция и теги (AND-логика)
   - поиск коллекций — по названию
8. *(в будущем)* Просмотр графа связей

### 4.2. Что юзер делает чаще всего?

**Добавляет материал.** Это критично для перформанса —
операции `INSERT` в `Material` должны быть быстрыми.

### 4.3. Редкие, но важные операции

- **Экспорт всех данных** — требование GDPR / 152-ФЗ для
  коммерческого продукта.
- **Удаление аккаунта** с каскадным удалением всех данных.
- **Бэкап БД** — на стороне сервера, не юзера.
- **Импорт из других сервисов** (Notion, Obsidian) — в roadmap.

## Блок 5. Критичность

### 5.1. Приоритеты приложения

1. Надёжность данных
2. Скорость работы
3. Приватность
4. Простота
5. Красота

### 5.2. Что если БД упадёт на час?

БД — **критичная зависимость**. Падение на час = недоступность сервиса.

Требуется:
- автоматический бэкап раз в сутки
- мониторинг доступности

### 5.3. Лимиты файлов

| Параметр | Значение |
|---|---|
| Размер одного файла | до **50 MB** |
| Общий объём на юзера (free) | **500 MB** |
| Разрешённые типы (документы) | `pdf`, `docx`, `rtf` — поиск по содержимому |
| Разрешённые типы (тексты) | `txt`, `md` — поиск по содержимому |
| Разрешённые типы (таблицы) | `xlsx`, `csv` — без поиска по содержимому |
| Разрешённые типы (изображения) | `jpg`, `jpeg`, `png` — без поиска по содержимому |
| Запрещённые типы | `exe`, `zip`, `mp4` (тяжёлое видео), все исполняемые |

### 5.4. Сколько материалов на юзера?

- **Free-тариф:** до **1 000 материалов** на юзера
- **PRO-тариф:** без лимита

Технических ограничений нет (БД держит миллионы),
лимит нужен для монетизации — чтобы активные юзеры
со временем переходили на PRO.

## Блок 6. Технические рамки

### 6.1. Платформы

- **MVP:** только web (адаптив под мобильные браузеры)
- **Roadmap:** PWA → нативная мобилка (React Native)
- **Десктоп:** не планируется

### 6.2. Хостинг

- **MVP:** VPS у Timeweb или Selectel (~500 ₽/мес)
- **При росте:** Yandex Cloud с managed PostgreSQL и Object Storage
- **AWS** не рассматривается из-за платёжных и юридических
  ограничений в РФ

### 6.3. Стадия проекта

**MVP с прицелом на коммерческий продукт.**

Модель монетизации (предположительно): **freemium**
(бесплатно до X материалов / Y MB, дальше подписка).
