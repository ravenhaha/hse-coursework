# Domain Document — «Омут памяти» (ранняя версия)

## 1. Общее описание

«Омут памяти» — это концепция цифровой системы для накопления, структурирования,
связывания и повторного использования знаний, заметок и учебных материалов пользователя.

В раннем понимании проект рассматривался не только как учебное хранилище,
но и как более широкая система личной памяти / персональной базы знаний.

---

## 2. Проблема

Современный пользователь хранит важную для себя информацию фрагментированно:
- в заметках,
- в документах,
- в папках на диске,
- в браузерных закладках,
- в мессенджерах,
- в облачных сервисах,
- в скриншотах и фотографиях.

Из-за этого возникают:
- потеря контекста;
- сложность поиска;
- дублирование информации;
- ощущение «я это уже видел, но не могу найти»;
- плохая повторная используемость знаний.

---

## 3. Цель системы

Создать систему, которая позволит пользователю:
- сохранять разные типы материалов;
- структурировать их по темам;
- размечать тегами;
- связывать между собой;
- возвращаться к знаниям в удобной форме;
- использовать накопленное как для хранения, так и для повторения.

---

## 4. Целевая аудитория (ранняя версия)

На раннем этапе целевая аудитория описывалась широко.

### Возможные пользователи
- студент;
- школьник;
- преподаватель;
- исследователь;
- автор / креатор;
- человек, ведущий личный архив знаний;
- любой пользователь, работающий с большим объёмом заметок и материалов.

### Замечание
Позже фокус проекта был сужен до более конкретного учебного сценария,
но в этом документе сохранена ранняя широкая рамка.

---

## 5. Основные сценарии использования

### 5.1. Сохранение материала
Пользователь добавляет:
- текстовую заметку;
- файл;
- ссылку;
- изображение;
- конспект;
- цитату.

### 5.2. Организация по темам
Пользователь раскладывает материалы:
- по коллекциям;
- по тегам;
- по темам;
- по связям с другими материалами.

### 5.3. Поиск и возврат к знаниям
Пользователь:
- ищет по названию;
- ищет по содержимому;
- переходит по связанным материалам;
- восстанавливает контекст темы.

### 5.4. Повторение и закрепление
Пользователь:
- отмечает важное;
- формирует карточки;
- повторяет трудные темы;
- видит, к чему стоит вернуться.

### 5.5. Публикация / совместное использование
В ранних гипотезах также допускалось:
- делиться коллекцией;
- открывать материалы по ссылке;
- делать подборки публичными.

## 6. Роли пользователей (ранняя рамка)

### 6.1. Гость
Может:
- открыть лендинг;
- посмотреть описание продукта;
- зарегистрироваться или войти.

### 6.2. Пользователь
Может:
- создавать коллекции;
- создавать материалы;
- загружать файлы;
- создавать теги;
- связывать материалы;
- использовать поиск;
- работать с личным пространством.

### 6.3. Администратор
В ранней версии допускалась административная роль для:
- модерации пользователей;
- просмотра статистики;
- разбора жалоб;
- технического администрирования.

### Комментарий
Позже продуктовый фокус был смещён в сторону личного пространства,
и админский слой перестал быть частью core-MVP.

## 7. Ранняя предметная модель

Ниже — ранний набор сущностей,
часть из которых позже была упрощена или отложена.

---

### 7.1. User
Пользователь системы.

Свойства:
- регистрируется;
- входит в систему;
- владеет своими материалами;
- владеет своими коллекциями;
- владеет своими тегами;
- может иметь публичный профиль или настройки видимости.

---

### 7.2. Collection
Коллекция материалов.

Раннее понимание:
- базовая тематическая единица;
- может быть плоской на MVP;
- может стать вложенной в будущих версиях;
- используется для группировки материалов.

---

### 7.3. Material
Базовая единица контента.

В раннем документе material понимался широко:
- текст;
- файл;
- ссылка;
- изображение;
- конспект;
- возможно, карточка или chunk как отдельный подтип.

---

### 7.4. Tag
Пользовательская метка.

Теги нужны для:
- категоризации;
- быстрого поиска;
- пересечения тем;
- фильтрации.

В ранних обсуждениях также допускались:
- цвет тега;
- группировка тегов;
- иерархия тегов.

---

### 7.5. MaterialLink
Ранняя идея отдельной связи между материалами.

Возможные типы связей:
- связано с;
- продолжение;
- часть темы;
- объясняет;
- противоречит;
- пример к;
- нужно перед изучением.

Эта сущность рассматривалась как основа для knowledge graph.

---

### 7.6. Thread / Path
Последовательность материалов по определённой учебной логике.

Например:
- подготовка к экзамену;
- путь по теме;
- цепочка повторения.

---

### 7.7. Card
Карточка для повторения.

Возможные поля:
- вопрос;
- ответ;
- источник;
- тема;
- сложность.

---

### 7.8. ReviewSchedule / RepeatPlan
Сущность для механики повторения.

Потенциально могла хранить:
- дату следующего повторения;
- интервал;
- число успешных повторов;
- сложность материала.

---

### 7.9. SharedCollection / PublicCollection
Идея публичного или совместного пространства.

Назначение:
- делиться подборкой материалов;
- давать доступ по ссылке;
- собирать открытые коллекции по теме.

---

## 8. Связи между сущностями (раннее представление)

### User → Collection
Один пользователь имеет много коллекций.

### User → Material
Один пользователь имеет много материалов.

### User → Tag
Один пользователь имеет много тегов.

### Collection → Material
Коллекция содержит материалы.

### Material ↔ Tag
Материал может иметь несколько тегов.

### Material ↔ Material
Материалы могут быть связаны через отдельную сущность link.

### Material → Card
Из материала могут порождаться карточки.

### User → ReviewSchedule
Пользователь может иметь персональный план повторения.

---

## 9. Раннее понимание графа

В этом документе граф считался одной из центральных идей системы.

### Что предполагалось
- пользователь видит сеть материалов;
- материалы соединены смысловыми связями;
- можно переходить по соседним узлам;
- граф помогает восстанавливать контекст темы.

### Возможные источники связей
- ручные связи пользователя;
- общие теги;
- общая тема;
- AI-определённая близость;
- принадлежность одной коллекции.

### Проблема ранней версии
Не было ясного ответа:
- какие связи считать каноническими;
- кто их создаёт;
- как не перегрузить пользователя.

Система должна была:
- быть веб-доступной;
- обеспечивать сохранность пользовательских данных;
- иметь понятную структуру;
- поддерживать рост объёма материалов;

- не требовать от пользователя сложного порога входа.
