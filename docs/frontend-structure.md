# 🏗️ Структура фронтенда

## 📁 Общая организация

Код в `src/` разложен **по типам файлов**:

```
src/
├── api/         ← модули запросов к бэкенду
├── assets/      ← статика
├── components/  ← React-компоненты
├── context/     ← глобальное состояние (React Context)
├── hooks/       ← кастомные хуки
├── layouts/     ← обёртки страниц
├── pages/       ← страницы (роуты)
├── styles/      ← глобальные стили
├── utils/       ← вспомогательные функции
├── App.jsx
└── main.jsx
```

Каждая папка отвечает за свой **технический слой** приложения.

---

## ⚠️ Замечание по структуре (для фронтенд-разработчика)

### Что сейчас

Используется подход **«организация по типам файлов»** (type-based / layered).
Все компоненты лежат в одной общей папке `components/`, все хуки — в `hooks/`,
все запросы — в `api/`, и т.д.

Это самый базовый подход, с которого обычно начинают учебные проекты.

### Почему он хуже современного подхода

Когда проект растёт, общие папки превращаются в **свалку**:

- В `components/` оказываются десятки несвязанных компонентов
  (`AuthForm`, `Sidebar`, `MaterialCard`, `PasswordInput`, `TagFilter`...).
- Связанный код одной фичи (например, авторизации) **разбросан по 5+ папкам**:
  `api/auth.js`, `context/AuthContext.jsx`, `components/Auth/*`,
  `hooks/useAuth.js`, `utils/passwordRules.js`.
- **Непонятно**, какой компонент к какой фиче относится и где он используется.
- **Сложно удалить фичу** — нужно вручную выискивать её файлы по всему проекту.
- Размыта граница между **переиспользуемым** кодом и кодом, нужным **только одной фиче**.

### Современная альтернатива: Feature-based структура

В индустрии (Redux Toolkit, Next.js, Bulletproof React, Feature-Sliced Design)
принято раскладывать код **по фичам**, а не по типам:

```
src/
├── features/        ← каждая фича в своей папке
│   ├── auth/        (api + components + hooks + utils фичи)
│   ├── materials/
│   └── tags/
├── pages/           ← тонкие страницы, только композиция
├── shared/          ← общий код (ui-примитивы, утилиты, api-клиент)
└── layouts/
```

**Главный принцип:** код, который меняется вместе, должен лежать вместе.

### Что это даёт

- Открыл `features/auth/` — видишь **всю** авторизацию в одном месте.
- Удалить фичу = удалить одну папку.
- Чёткая граница: `shared/` — переиспользуемое, `features/` — локальное.
- Структура масштабируется на сотни компонентов без бардака.

### Рекомендация

При активном росте проекта **рекомендуется постепенный переезд**
на feature-based структуру — фича за фичей, без остановки разработки.

📚 Подробнее: [Feature-Sliced Design](https://feature-sliced.design/),
[Bulletproof React](https://github.com/alan2207/bulletproof-react).
