# Database Model (Current) / Актуальная модель базы данных

Этот документ описывает текущее состояние модели данных с учётом актуальной реализации backend и changelog.

## Общий принцип
База данных проекта хранит:
- пользователей;
- способы аутентификации;
- коллекции;
- материалы;
- теги;
- связи материалов с тегами.

Граф в текущем MVP не хранится как отдельная таблица. Он формируется из иерархии коллекций и материалов.

---

## 1. Таблица `users`

Назначение: хранение пользовательских аккаунтов.

### Основные поля
- `user_id` — первичный ключ;
- `user_email` — email пользователя;
- `display_name` — отображаемое имя;
- `avatar_path` — относительный путь к аватару, если он используется;
- `is_active` — флаг активности аккаунта;
- `created_at` — дата создания.

### Ограничения
- `user_email` уникален.

### Комментарий
Пароль не хранится в таблице `users`. Локальный пароль хранится в `auth_accounts` для записи с `provider='email'`.

---

## 2. Таблица `auth_accounts`

Назначение: хранение способов входа пользователя.

### Основные поля
- `auth_id` — первичный ключ;
- `user_id` — внешний ключ на `users`;
- `provider` — тип провайдера (`email`, `vk`, `yandex`);
- `provider_user_id` — идентификатор пользователя у провайдера;
- `password_hash` — хэш пароля для `provider='email'`, иначе `NULL`;
- `created_at` — дата создания.

### Ограничения
- `UNIQUE (provider, provider_user_id)`.

### Бизнес-смысл
Один пользователь может иметь несколько способов входа:
- email/password;
- Yandex OAuth;
- VK OAuth.

---

## 3. Таблица `collections`

Назначение: хранение пользовательских коллекций.

### Основные поля
- `collection_id` — первичный ключ;
- `user_id` — владелец коллекции;
- `parent_id` — ссылка на родительскую коллекцию, `NULL` для корневых;
- `collection_name` — название коллекции;
- `icon` — emoji-иконка коллекции, nullable;
- `created_at` — дата создания.

### Ограничения
- `UNIQUE (user_id, parent_id, collection_name)`.

### Особенности
- поддерживается вложенность коллекций;
- иконка хранится как строка с emoji, а не как файл.

---

## 4. Таблица `materials`

Назначение: хранение текстовых и файловых материалов.

### Основные поля
- `material_id` — первичный ключ;
- `collection_id` — внешний ключ на `collections`;
- `material_name` — название материала;
- `source_type` — тип материала (`text` или `file`);
- `text_content` — содержимое текстовой заметки;
- `file_path` — путь к загруженному файлу;
- `file_size` — размер файла в байтах;
- `extracted_text` — текст, извлечённый из файла для поиска;
- `created_at` — дата создания.

### Ограничения
Используется `CHECK`-ограничение согласованности содержимого:

- если `source_type='text'`:
  - `text_content` обязательно;
  - `file_path IS NULL`;
  - `file_size IS NULL`;
  - `extracted_text IS NULL`;

- если `source_type='file'`:
  - `file_path` обязательно;
  - `file_size` обязательно;
  - `text_content IS NULL`;
  - `extracted_text` может быть `NULL`, если парсер не сработал или формат не поддерживается.

### Особенности
- `material_name` используется как пользовательское имя материала;
- отдельное поле `original_name` в текущей модели не используется;
- отдельное поле `mime_type` в MVP не используется.

---

## 5. Таблица `tags`

Назначение: хранение персональных тегов пользователя.

### Основные поля
- `tag_id` — первичный ключ;
- `user_id` — владелец тега;
- `tag_name` — название тега;
- `created_at` — дата создания.

### Ограничения
- `UNIQUE (user_id, tag_name)`.

### Особенности
- при регистрации пользователю могут создаваться дефолтные теги;
- дефолтные теги являются обычными тегами пользователя;
- пользователь может их переименовывать или удалять.

---

## 6. Таблица `material_tags`

Назначение: реализация связи многие-ко-многим между материалами и тегами.

### Основные поля
- `material_id` — внешний ключ на `materials`;
- `tag_id` — внешний ключ на `tags`.

### Ограничения
- уникальность пары `material_id + tag_id` должна исключать дублирование связи.

### Особенности
- при удалении материала удаляются его связи с тегами;
- при удалении тега удаляются его связи с материалами;
- сами материалы и теги при этом не удаляются друг через друга.

---

## Связи между таблицами

- `users → collections` = 1:M
- `users → tags` = 1:M
- `users → auth_accounts` = 1:M
- `collections → collections` = 1:M
- `collections → materials` = 1:M
- `materials ↔ tags` = M:N через `material_tags`

---

## Каскадные удаления

### Гарантированно поддерживаются
- удаление пользователя приводит к удалению связанных сущностей пользователя;
- удаление материала удаляет его связи в `material_tags`;
- удаление тега удаляет его связи в `material_tags`.

### Важно
Для удаления коллекций и вложенных структур должна сохраняться согласованность данных. Реализация может обеспечиваться:
- на уровне каскадных связей,
- на уровне сервисной логики,
- либо комбинированно.

---

## Поиск по материалам

Поиск материалов в текущей реализации работает по:
- `material_name`;
- `text_content`;
- `extracted_text`.

Дополнительно могут использоваться фильтры:
- `collection_id`;
- `tag_ids` с AND-логикой.

---

## Защита от race condition

Для операций, завязанных на `UNIQUE`-ограничения, текущая реализация использует:
- предварительную проверку;
- обработку `IntegrityError`.

Это особенно важно для:
- тегов;
- сущностей с уникальными именами;
- параллельных запросов и двойных кликов.
