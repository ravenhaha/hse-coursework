# API Map (Current) / Карта текущего REST API

Этот документ описывает актуальную карту backend-маршрутов на уровне текущей реализации.

## Общие правила

### Формат API
- REST API
- JSON для стандартных запросов и ответов
- multipart/form-data для загрузки файлов

### Доступ
Все защищённые маршруты доступны только аутентифицированному пользователю.

### Cookie-стратегия
Текущая auth-реализация использует отдельные cookies:
- access token;
- refresh token;
- csrf token.

### CSRF
Для mutating-запросов (`POST`, `PUT`, `PATCH`, `DELETE`) используется заголовок:
- `X-CSRF-Token`

Проверка CSRF:
- не применяется к safe-методам (`GET`, `HEAD`, `OPTIONS`);
- не применяется к login/register/refresh/logout;
- не применяется к OAuth-роутам;
- управляется конфигом окружения.

## 1. Auth

### `POST /auth/register`
Регистрация пользователя по email и паролю.

Особенности:
- может включать автологин после успешной регистрации;
- при первой регистрации создаются дефолтные теги пользователя.

### `POST /auth/login`
Локальный вход пользователя.

### `POST /auth/logout`
Выход пользователя и завершение сессии.

### `POST /auth/refresh`
Обновление access-сессии по refresh-механизму.

### `GET /auth/me`
Получение данных текущего пользователя.

### `GET /auth/yandex`
Начало OAuth-аутентификации через Yandex.

### `GET /auth/yandex/callback`
Обработка возврата от Yandex OAuth.

### `GET /auth/vk`
Начало OAuth-аутентификации через VK.

### `GET /auth/vk/callback`
Обработка возврата от VK OAuth.

Комментарий:
- VK-поток может требовать отдельной настройки secret в окружении;
- при неполной конфигурации реализация может возвращать `501 Not Implemented`.

## 2. Collections

### `GET /collections`
Получение списка коллекций пользователя.

### `GET /collections/search`
Поиск коллекций по названию.

### `GET /collections/{collection_id}`
Получение одной коллекции.

### `POST /collections`
Создание коллекции.

Поддерживаемые поля на уровне бизнес-смысла:
- название;
- родительская коллекция;
- emoji-иконка.

### `PATCH /collections/{collection_id}`
Изменение коллекции.

Поддерживаемые сценарии:
- переименование;
- смена `parent_id`;
- изменение `icon`.

### `DELETE /collections/{collection_id}`
Удаление коллекции.

---

## 3. Materials

### `GET /materials`
Получение списка материалов.

Обычно используется для:
- списка материалов в коллекции;
- фильтрации по `collection_id`.

### `GET /materials/search`
Поиск материалов.

Текущая логика поиска:
- подстрока в `material_name`;
- подстрока в `text_content`;
- подстрока в `extracted_text`.

Фильтры:
- `collection_id`;
- `tag_ids` с AND-логикой.

### `GET /materials/{material_id}`
Получение одного материала.

### `POST /materials/text`
Создание текстового материала.

### `POST /materials/file`
Создание файлового материала.

Особенности:
- файл загружается отдельно;
- backend сохраняет `file_path`, `file_size`;
- при возможности извлекается `extracted_text`.

### `PATCH /materials/{material_id}`
Изменение материала.

Типовые сценарии:
- переименование;
- редактирование текста;
- перемещение в другую коллекцию.

### `DELETE /materials/{material_id}`
Удаление материала.

---

## 4. Tags

### `GET /tags`
Получение списка тегов пользователя.

### `GET /tags/{tag_id}`
Получение одного тега.

### `POST /tags`
Создание тега.

### `PATCH /tags/{tag_id}`
Переименование тега.

### `DELETE /tags/{tag_id}`
Удаление тега.

### `GET /tags/materials/{material_id}`
Получение тегов, назначенных материалу.

### `POST /tags/materials/{material_id}`
Назначение одного тега материалу.

### `DELETE /tags/materials/{material_id}/{tag_id}`
Снятие одного тега с материала.

### `PUT /tags/materials/{material_id}`
Bulk-замена полного набора тегов материала.

## Основные коды ответов

- `200 OK` — успешное получение или обновление
- `201 Created` — объект создан
- `204 No Content` — объект удалён
- `400 Bad Request` — некорректный запрос
- `401 Unauthorized` — требуется аутентификация
- `403 Forbidden` — доступ запрещён
- `404 Not Found` — объект не найден
- `409 Conflict` — конфликт уникальности или состояния
- `413 Payload Too Large` — слишком большой файл
- `422 Unprocessable Entity` — ошибка валидации
- `501 Not Implemented` — функциональность не настроена в окружении

## Что не считать частью обязательного текущего API

Если в старых материалах встречаются маршруты для:
- карточек;
- тренировок;
- попыток;
- повторений;
- админки;
- техподдержки;
- совместных коллекций;

их не нужно считать частью обязательного API текущего MVP.
