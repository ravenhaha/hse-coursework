# Backend Architecture / Архитектура серверной части

## Общая идея
Серверная часть проекта отвечает за:
- аутентификацию и авторизацию;
- хранение данных;
- работу с коллекциями;
- работу с материалами;
- работу с тегами;
- файловую обработку;
- подготовку данных для графовой визуализации.

## Технологическая рамка
Актуальная backend-реализация использует:
- FastAPI как web-framework;
- PostgreSQL как основную БД;
- Alembic для миграций;
- файловое хранилище для загруженных материалов;
- гибридный подход к доступу к БД:
  - API работает асинхронно;
  - миграции выполняются синхронно.

## Архитектурный стиль
Используется слоистая архитектура.

## Основные слои

### 1. API layer
Отвечает за:
- HTTP-маршруты;
- валидацию входных данных;
- схемы запросов и ответов;
- интеграцию зависимостей FastAPI;
- возврат кодов ошибок и JSON-ответов.

### 2. Service layer
Отвечает за бизнес-логику:
- регистрацию;
- логин;
- проверки доступа;
- создание и редактирование коллекций;
- создание и редактирование материалов;
- работу с тегами;
- поиск;
- подготовку структуры для графа.

### 3. Data access layer
Отвечает за:
- CRUD-операции;
- работу с ORM-моделями;
- запросы к PostgreSQL;
- выборки, фильтры, проверки уникальности.

### 4. Infrastructure layer
Включает:
- конфигурацию приложения;
- cookies и токены;
- CSRF-защиту;
- файловое хранилище;
- парсинг загруженных файлов;
- обработку исключений;
- миграции и окружение.

## Обработка запроса

1. Клиент отправляет HTTP-запрос.
2. API-слой принимает запрос.
3. Проверяется аутентификация.
4. Для mutating-запросов проверяется CSRF, если защита включена.
5. Выполняется валидация входных данных.
6. Запрос передаётся в сервисный слой.
7. Сервисный слой обращается к слою доступа к данным.
8. Выполняется чтение или изменение данных.
9. При необходимости выполняется работа с файлами.
10. Клиент получает JSON-ответ или ошибку.

## Аутентификация
Текущая реализация поддерживает:
- локальную регистрацию по email и паролю;
- локальный логин;
- refresh-цикл;
- OAuth через внешних провайдеров;
- HttpOnly-cookie стратегию для токенов;
- CSRF double-submit pattern.

## Контроль доступа
Каждый пользователь может выполнять операции только над своими:
- коллекциями;
- материалами;
- тегами;
- auth-привязками.

Проверка принадлежности объекта пользователю является обязательной частью серверной логики.

## Работа с файлами
Backend отвечает за:
- приём загрузок;
- хранение пути к файлу;
- сохранение размера файла;
- извлечение текста из поддерживаемых форматов, если доступен парсер;
- использование извлечённого текста в поиске.

## Работа с графом
Backend не хранит отдельный сложный граф знаний.
Он подготавливает данные для визуализации структуры, построенной на основе:
- иерархии коллекций;
- принадлежности материалов коллекциям.

## Практический акцент текущей версии
В текущей курсовой версии основной акцент делается на:
- backend-части;
- модели данных;
- API;
- аутентификации;
- согласованности связей;
- корректной работе с БД и файлами.
