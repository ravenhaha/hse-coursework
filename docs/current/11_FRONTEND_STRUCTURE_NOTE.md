# Frontend Structure Note / Заметка по структуре фронтенда

Этот документ описывает текущую организацию frontend-кода и рекомендации по её развитию.

## Текущее устройство

Код в `src/` организован по типам файлов:

```text
src/
├── api/
├── assets/
├── components/
├── context/
├── hooks/
├── layouts/
├── pages/
├── styles/
├── utils/
├── App.jsx
└── main.jsx
```

## Что означает текущий подход
Это type-based / layered organization:
- все компоненты лежат в одном месте;
- все API-модули лежат в одном месте;
- все хуки лежат в одном месте;
- все утилиты лежат в одном месте.

Такой подход допустим для учебного проекта и удобен на старте.

## Проблемы подхода при росте проекта
Когда проект разрастается, появляются типичные сложности:
- папка `components/` становится слишком большой;
- код одной фичи оказывается размазан по нескольким каталогам;
- сложнее понять, что к чему относится;
- удаление фичи требует поиска файлов по всему проекту;
- хуже видна граница между shared-кодом и feature-кодом.

## Рекомендуемое направление развития
При дальнейшем росте проекта рекомендуется постепенный переход к feature-based структуре.

Пример:

```text
src/
├── features/
│   ├── auth/
│   ├── collections/
│   ├── materials/
│   └── tags/
├── pages/
├── shared/
└── layouts/
```

## Принцип feature-based подхода
Код, который меняется вместе, должен лежать вместе.

Это даёт:
- лучшую локализацию логики;
- более понятную структуру;
- упрощение поддержки;
- удобство удаления и рефакторинга фич.

## Практическая рекомендация
Не требуется переписывать весь фронтенд сразу.

Оптимальный путь:
- оставлять текущую структуру как рабочую;
- новые крупные блоки переносить фича за фичей;
- вынести действительно общий код в `shared`.
