## [v0.5] — Полноценная реализация тегов

**Дата:** _уточнить по истории коммитов_

### Реализовано
- CRUD тегов: создание, переименование, удаление в рамках пользователя.
- Уникальность имени тега в рамках пользователя через `UNIQUE (user_id, tag_name)`.
- Привязка тегов к материалам через M2M-таблицу `material_tags`.
- Каскадное удаление связей:
  - удалили материал → пропали его связи с тегами;
  - удалили тег → пропали его связи с материалами;
  - сами материалы и теги при этом не удаляются друг через друга.
- Поддержаны три сценария работы со связями:
  - `POST /tags/materials/{material_id}` — добавить один тег;
  - `DELETE /tags/materials/{material_id}/{tag_id}` — убрать один тег;
  - `PUT /tags/materials/{material_id}` — bulk-замена всего набора тегов.
- Дефолтные теги создаются при регистрации:
  - `лекция`
  - `семинар`
  - `домашка`
  - `экзамен`
  - `конспект`
  - `формула`
  - `важно`
  - `повторить`

### Уточнение
- На этом этапе не вводилась новая отдельная доменная сущность сверх базовой модели тегов;
  была доведена до полноценной реализации уже заложенная ранее tag-система.
- Дефолтные теги являются обычными пользовательскими тегами.
  Пользователь может их переименовывать и удалять как любые другие свои теги.

### Защита от race condition
Операции, завязанные на `UNIQUE`-ограничения, защищены через:
- предварительную проверку;
- `try/except IntegrityError`.

Это страхует от:
- двойного клика;
- параллельных вкладок;
- одновременных запросов.

### Решено не добавлять пока
- цвет тега;
- глобальные shared-теги;
- иерархию тегов.

## [v0.4] — Иконки коллекций и завершение auth-стека

**Дата:** _уточнить по истории коммитов_

### Добавлено
- Поле `Collection.icon` — emoji-иконка коллекции (`String(10)`, nullable).

### Почему выбраны emoji, а не картинки
- не требуют хранения файлов;
- почти ничего не весят;
- рендерятся мгновенно;
- соответствуют UX-паттерну популярных продуктов.

### Реализовано
- локальная регистрация по email + password с автологином;
- локальный логин;
- refresh-цикл;
- OAuth через Yandex;
- OAuth через VK;
- HttpOnly-cookie стратегия:
  - access;
  - refresh;
  - csrf.
- CSRF double-submit pattern:
  - `csrf_token` кладётся в отдельную cookie;
  - фронт отправляет его в `X-CSRF-Token`;
  - сервер сверяет cookie и header;
  - safe-методы не проверяются;
  - auth-роуты и OAuth-роуты исключены;
  - защита управляется env-флагом.
- `Argon2` для хеширования паролей с возможностью auto-rehash.
- Дефолтные теги создаются при первой регистрации любым способом входа.

### Уточнение по auth-слою
- Access и refresh используются как основа cookie-based auth-сессии.
- CSRF-токен вынесен отдельно, чтобы mutating-запросы проверялись через double-submit pattern.
- При OAuth-потоках и auth-роутах действуют исключения, потому что в этих сценариях обычная CSRF-проверка работает иначе, чем для прикладных CRUD-запросов.

### Примечание по VK
VK OAuth реализован, но требует корректной настройки окружения.
Без нужного секрета может возвращаться `501`.

### Решено не добавлять
- загружаемую обложку коллекции;
- отдельное поле `notes` / `description` у материала;
- batch-upload эндпоинт как обязательную backend-операцию.

### Почему не делали отдельный batch-upload endpoint
В текущей логике фронт может отправлять несколько обычных запросов на одиночный upload endpoint.
Это даёт:
- независимые транзакции;
- частичный успех;
- более простую backend-реализацию;
- более удобный контроль ошибок по каждому файлу.

## [v0.3] — Метаданные файлов и поиск по содержимому

**Дата:** _уточнить по истории коммитов_

### Добавлено
- `Material.file_size` — размер файла в байтах.
- `Material.extracted_text` — текст, извлечённый из PDF / DOCX / TXT / MD / RTF и используемый в поиске.

### Зачем это понадобилось
- `file_size` нужен для отображения информации о файле и как база для возможных будущих ограничений по storage.
- `extracted_text` нужен для поиска по содержимому файловых материалов, а не только по имени файла.
- `extracted_text` хранится отдельно от `text_content`, потому что:
  - `text_content` — это нативное содержимое текстовой заметки;
  - `extracted_text` — это вторичный поисковый слой для файлового материала.

### Изменено
Расширено `CHECK`-ограничение согласованности содержимого:

#### Для `source_type='text'`
- `text_content` обязателен;
- file-поля должны быть `NULL`.

#### Для `source_type='file'`
- `file_path` обязателен;
- `file_size` обязателен;
- `text_content` должен быть `NULL`;
- `extracted_text` может быть `NULL`.

### Поиск
`search_materials` теперь ищет подстроку в:
- `material_name`;
- `text_content`;
- `extracted_text`.

Поддерживаются фильтры:
- `collection_id`;
- `tag_ids` с AND-логикой.

### Решено не добавлять
- `Material.original_name`;
- `Material.mime_type`.

### Почему не добавили `Material.original_name`
Отдельное хранение оригинального имени файла было сочтено избыточным, потому что:
- пользователь всё равно ориентируется на `material_name`;
- пользователь может переименовать материал под себя;
- лишнее поле усложняет модель без критической пользы для MVP.

### Почему не добавили `Material.mime_type`
Для текущего MVP отдельное поле MIME-типа не является обязательным:
- логика хранения и поиска от него не зависит;
- для базового сценария хватает пути к файлу и типа источника.

### Отложено
- `User.plan`;
- `User.storage_used`;
- PostgreSQL full-text индексы через `tsvector`.

## [v0.2] — Обновление логической модели и OAuth-основа

**Дата:** 11.05.26

### Добавлено
- Таблица `auth_accounts` для поддержки нескольких способов входа.
- `User.is_active`.
- `User.avatar_path`.

### Смысл `auth_accounts`
Таблица `auth_accounts` позволила отвязать способы входа от основной пользовательской записи.

Один пользователь может:
- иметь только OAuth-вход;
- иметь email/password;
- иметь email/password и дополнительно OAuth;
- не дублировать auth-данные в `users`.

### Изменено
- `User.hashed_password` убран из таблицы `users`.
- Пароль теперь хранится в `auth_accounts.password_hash` только для записи с `provider='email'`.

### Исправлено
- колонка называется `provider`, а не `provider_auth`;
- используется `UNIQUE (provider, provider_user_id)`.

### Отложено
- `Material.mime_type`;
- `Material.is_important`;
- `User.plan`;
- `User.storage_used`;
- `User.updated_at`.

## Общий итог эволюции
Текущая реализация ушла дальше первоначальной базовой схемы за счёт:
- вынесения auth-данных в отдельную таблицу;
- поддержки OAuth;
- emoji-иконок коллекций;
- file metadata;
- извлечённого текста;
- полноценной tag-системы;
- cookie + csrf auth-стека.

При этом продуктовая суть проекта не изменилась:
это всё ещё система для персональной организации учебных материалов.
